# Motion Safe Workflow — MALEA Landing

> Система правил для безопасного итерационного внедрения анимации на лендинге MALEA.
> Motion — отдельный слой дизайна. Внедряется постепенно, по устройствам, без смешивания с layout и responsive.

---

## 1. Общий принцип

- Motion является частью дизайна лендинга, а не декоративным эффектом.
- Motion внедряется постепенно.
- Нельзя реализовывать всю анимацию сразу.
- Нельзя делать универсальную анимацию для всех устройств.
- Нельзя смешивать motion и layout без необходимости.
- Motion должен быть отдельным слоем, а не хаотичными правками в существующих responsive-блоках.

---

## 2. Порядок устройств

Работать строго в таком порядке:

1) **Desktop**;
2) **Tablet landscape**;
3) **Tablet portrait**;
4) **Mobile**.

Пока идёт работа с **desktop**:

- tablet landscape не трогать;
- tablet portrait не трогать;
- mobile не трогать.

Пока идёт работа с **tablet landscape**:

- desktop не ломать;
- tablet portrait не трогать;
- mobile не трогать.

Пока идёт работа с **tablet portrait**:

- учитывать [`css/tablet-portrait-overrides.css`](../css/tablet-portrait-overrides.css);
- не ломать уже сделанную vertical tablet вёрстку;
- desktop / tablet landscape / mobile не трогать.

Пока идёт работа с **mobile**:

- не переносить desktop-анимацию напрямую;
- приоритет: читаемость, отсутствие лагов, отсутствие горизонтального скролла.

---

## 3. Desktop motion workflow

Если пользователь даёт задачу «сделать анимацию на desktop», Roo должен:

- работать только с desktop-профилем;
- не менять планшетные и мобильные media queries;
- использовать отдельные desktop-scoped классы / файлы;
- не вносить `transform` / `opacity` / `blur` в глобальные селекторы;
- не править `body`, `section`, `h1`, `h2`, `p`, `.screen` глобально;
- использовать точные section id и motion-классы;
- делать сначала один переход / один экран / один приём;
- после проверки распространять приём на группу экранов.

---

## 4. Screen correction workflow

Если пользователь говорит «скорректировать экран» в процессе анимации, Roo должен:

- сначала определить устройство: desktop / tablet landscape / tablet portrait / mobile;
- определить конкретный экран по section id;
- работать только с этим экраном;
- работать только с этим breakpoint / profile;
- не менять соседние экраны;
- не менять другие устройства;
- не делать глобальный refactor;
- не использовать `margin` / `transform` как костыль без объяснения;
- сначала диагностировать причину;
- потом делать минимальную правку;
- потом проверить визуально и через computed styles.

---

## 5. Разрешённые файлы для будущей desktop-анимации

Будущая desktop-анимация должна по возможности жить отдельно:

- [`css/motion-desktop.css`](../css/motion-desktop.css)
- [`js/motion-desktop.js`](../js/motion-desktop.js)

Если этих файлов ещё нет, не создавать их без отдельной команды пользователя.

Не добавлять desktop motion в:

- [`css/tablet-portrait-overrides.css`](../css/tablet-portrait-overrides.css);
- legacy TP-блоки в [`css/styles.css`](../css/styles.css);
- mobile media queries;
- tablet landscape media queries.

---

## 6. Запрещённые действия без прямой команды пользователя

- делать анимацию сразу для всех устройств;
- менять tablet / mobile во время desktop-задачи;
- удалять legacy TP-блоки;
- подключать [`css/tablet-portrait-overrides.clean-candidate.css`](../css/tablet-portrait-overrides.clean-candidate.css);
- переписывать [`css/styles.css`](../css/styles.css);
- менять структуру HTML без необходимости;
- трогать player / video / carousel / menu без отдельной причины;
- делать publish;
- переключаться в main;
- делать merge / reset / rebase / clean.

---

## 7. Git safety

- Все обычные правки только в dev.
- main считается production.
- Production не трогать без прямой команды пользователя «опубликуй».
- После маленькой завершённой задачи можно использовать safe-save, если пользователь не запретил.
- Перед началом задачи проверять текущую ветку и git status.
- Если есть неожиданные незакоммиченные изменения — остановиться и сообщить пользователю.

---

## 8. Definition of Done для любой motion-задачи

Каждая задача считается завершённой только если Roo в отчёте указал:

- какой экран / экраны затронуты;
- какой тип устройства затронут;
- какие файлы изменены;
- какие селекторы / классы добавлены или изменены;
- какие устройства **НЕ** трогались;
- что проверено визуально;
- что проверено через computed styles;
- есть ли риск для соседних экранов;
- был ли safe-save;
- production / main не затронуты.
