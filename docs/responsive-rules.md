# Responsive Rules — MALEA

> Система правил, брейкпоинтов, запрещённых изменений и чеклист верификации.
> Нарушение любого правила требует письменного обоснования в PR.

---

## 1. Breakpoint system

Строго фиксированная система брейкпоинтов. Любое отклонение требует утверждения.

| Название | Диапазон | Типовые устройства |
|---|---|---|
| **Desktop** | ≥ 1101px | Ноутбуки, десктопы (1440px, 1366px) |
| **Tablet** | 761px – 1100px | iPad Pro portrait (1024px), iPad Air portrait (820px), iPad Mini portrait (768px) |
| **Mobile** | 521px – 760px | Крупные телефоны landscape, небольшие планшеты |
| **Small mobile** | ≤ 520px | iPhone 11/13/15 (390–414px), Samsung Galaxy |
| **Very small** | ≤ 380px | iPhone SE, Galaxy S24 compact |

> **Правило 1.1:** Запрещено добавлять брейкпоинты вне этого списка без утверждения в [`responsive-work-plan.md`](responsive-work-plan.md).

> **Правило 1.2:** Desktop-first наследуется от текущей архитектуры. Все mobile-правила — `max-width`.  
> Для tablet — `min-width: 761px` в комбинации с `max-width: 1100px`.

> **Правило 1.3:** `min-width` используется только для tablet-слоя (Stage 6) и desktop-улучшений. Не использовать `min-width` для управления mobile layout.

---

## 2. Обязательный принцип: единый responsive-layer

**Не чинить mobile хаотично внутри секций.**

Все новые adaptive-правки добавлять **только** в финальный блок в конце [`css/styles.css`](../css/styles.css):

```css
/* =========================================================
   FINAL RESPONSIVE SYSTEM — tablet / mobile
   All new responsive fixes must stay here.
   ========================================================= */
```

> **Правило 2.1:** На этапе подготовки (Stage 0) этот блок **НЕ добавлять** в [`css/styles.css`](../css/styles.css) без подтверждения. Только описать его规格 здесь.

> **Правило 2.2:** Начиная с Stage 1, все responsive-изменения пишутся строго внутри этого блока.

> **Правило 2.3:** После завершения всех Stage (1–6) существующие разрозненные media-запросы в [`css/styles.css`](../css/styles.css) должны быть перенесены в этот блок (консолидация) — отдельной задачей.

## 3. Запрещённые изменения

### 3.1 Категорически запрещено без отдельного задания

| № | Что запрещено | Почему |
|---|---|---|
| ❌ 3.1.1 | Менять desktop-версию (стили вне media-запросов для >1100px) | Desktop — базовая версия; её изменение требует отдельного дизайн-ревью |
| ❌ 3.1.2 | Менять HTML-структуру секций | Все id, классы и data-атрибуты используются в CSS и JS |
| ❌ 3.1.3 | Менять JS | Любое изменение JS требует отдельного тестирования |
| ❌ 3.1.4 | Менять iframe src или переносить iframe | Видео- и аудио-встраивания привязаны к конкретным источникам |
| ❌ 3.1.5 | Менять audio player logic | `#player`, `.track-grid`, `.player-cd-wrap` — всё завязано на JS (`playNewTrack()`, `togglePlay()`) |
| ❌ 3.1.6 | Менять video modal logic | `#videoModal`, `.vmodal-overlay`, `.video-modal-backdrop` — JS-функции `openVideoModal()`, `closeVideoModal()` |
| ❌ 3.1.7 | Менять mobile menu | `#mobileMenu`, `.nav-mobile-panel`, `.nav-mobile-open` — отдельная задача |
| ❌ 3.1.8 | Менять форму заявки | `#modalOverlay`, `.modal-overlay`, `.modal-box` — JS-функции `openModal()`, `closeModal()` |
| ❌ 3.1.9 | Удалять существующие классы (`data-screen-label`, `.screen`, id селекторов) | Нарушит dev-инструменты, карту экранов и CSS-стили |
| ❌ 3.1.10 | Удалять CSS-анимации (keyframes) | `cd-spin`, `wf-pulse`, `sensMarquee`, `hint-pulse` — часть бренда |
| ❌ 3.1.11 | Добавлять глобальные mobile-правила на `h2`, `p`, `.btn`, `.screen h2`, `body`, `section` без отдельного подтверждения | Все responsive-правки должны быть scoped по ID секции (`#hero`, `#identity`, и т.д.) или точному классу (`.identity-headline`, `.philosophy-title`, и т.д.) — широкие селекторы создают неявные каскадные конфликты |

### 3.2 Требуют осторожности

| № | Что | Условие |
|---|---|---|
| ⚠️ 3.2.1 | Изменение `z-index` | Только с проверкой всей цепочки наложения. Не использовать `!important` |
| ⚠️ 3.2.2 | Добавление `position: fixed` | Только после теста на мобильных Safari и Chrome |
| ⚠️ 3.2.3 | Изменение `overflow` на `.screen` | Может сломать скролл контейнера |
| ⚠️ 3.2.4 | Изменение `min-height: 100vh` / `height: 100vh` | **Full-screen sections:** использовать fallback `min-height: 100vh; min-height: 100svh;` — не заменять все `100vh` на `100dvh` слепо. **Fixed overlays / modals:** использовать `height: 100dvh; max-height: 100dvh;`. Каждое изменение — точечная правка с проверкой контекста. |
| ⚠️ 3.2.5 | Изменение `.photo-carousel` | JS-карусель рассчитывает позиции по DOM-дереву, любое изменение структуры сломает инфинитную прокрутку |

## 4. Проверка после каждого этапа

### 4.1 Разрешения для проверки

| Категория | Разрешения |
|---|---|
| **Desktop** | 1440px, 1366px |
| **Tablet** | 1024px, 820px, 768px |
| **Mobile** | 430px, 390px, 375px, 360px |

### 4.2 Что проверять

- [ ] Нет горизонтального скролла (`overflow-x` body).
- [ ] Заголовки не ломают экран (не переносятся некорректно, не перекрывают соседние элементы).
- [ ] Кнопки читаемы и имеют tap target ≥ 44px.
- [ ] Видео не ломается (iframe сохраняет пропорции).
- [ ] Модалки работают (видео-модалка, форма).
- [ ] Меню работает (десктопное + мобильное, если активировано).
- [ ] Форма открывается и закрывается.
- [ ] Console без ошибок.
- [ ] Desktop (1101px+) не изменился.

## 5. Где писать новые правила

В конце [`css/styles.css`](../css/styles.css) должен быть один главный блок:

```css
/* =========================================================
   FINAL RESPONSIVE SYSTEM — tablet / mobile
   All new responsive fixes must stay here.
   ========================================================= */
```

На этом этапе **НЕ добавлять** сам блок в CSS без подтверждения. Только описать, что его нужно будет добавить на следующем этапе (Stage 1).

Порядок внутри блока:

```css
/* =========================================================
   FINAL RESPONSIVE SYSTEM — tablet / mobile
   ========================================================= */

/* ─── Tablet: 761px–1100px ─── */
@media (min-width: 761px) and (max-width: 1100px) {
  /* Tablet-specific overrides */
}

/* ─── Mobile: 521px–760px ─── */
@media (max-width: 760px) {
  /* Mobile-specific overrides */
}

/* ─── Small mobile: ≤520px ─── */
@media (max-width: 520px) {
  /* Small mobile overrides */
}

/* ─── Very small: ≤380px ─── */
@media (max-width: 380px) {
  /* Very small mobile overrides */
}
```

> **Правило 5.1:** Все responsive-изменения — только в этом блоке.
> **Правило 5.2:** Не добавлять новые media-запросы вне этого блока.
> **Правило 5.3:** Порядок брейкпоинтов внутри блока: от большего к меньшему (desktop-first).

## 6. Целевые метрики качества

| Метрика | Цель |
|---|---|
| Lighthouse Mobile Performance | ≥ 85 |
| Lighthouse Mobile Accessibility | ≥ 95 |
| Время загрузки (3G, mobile) | ≤ 3.5s |
| First Contentful Paint (mobile) | ≤ 1.8s |
| Отсутствие горизонтального скролла на всех проверяемых разрешениях | 100% |
| Количество `!important` в CSS | 0 (после рефакторинга) |
| Console без ошибок на всех проверяемых разрешениях | ✅ |

## 7. Процесс внесения изменений

```
1. Выбрать Stage из [responsive-work-plan.md](responsive-work-plan.md).
2. Создать ветку: responsive/stage-N-name.
3. Внести изменения только внутри блока FINAL RESPONSIVE SYSTEM в [`css/styles.css`](../css/styles.css).
4. Пройти чеклист (раздел 4).
5. Проверить, что изменения не затронули desktop (1101px+).
6. Создать PR с описанием изменений.
7. Визуальное ревью на 4+ устройствах из раздела 4.1.
8. Мёрж только после прохождения всех проверок.
```
